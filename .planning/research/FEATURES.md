# Feature Landscape

**Domain:** AI Chat File Operation Recording (Electron + Vue 3 + TypeScript)
**Researched:** 2026-05-25

## Table Stakes

Features users expect when an AI tells them it changed files. Missing = user cannot trust or verify what the AI did.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| FileOperationRecord type | Without a typed record, every layer must guess at the shape of data. Users expect structured, consistent information about each change. | Low | New interface in `src/entityTypes/`. Fields: id, toolName, filePath, operationType (create/overwrite/edit), timestamp, success, errorMessage, conversationId, skillContext. Immutable pattern (readonly). |
| Record emission after each mutation tool | Users expect that when the AI says "I wrote X", a record appears immediately. Silence after a write = broken trust. | Low | Intercept in `ToolExecutor.executeFileTool()` after `FileToolService.execute()` returns. Emit record for `file_write` and `file_edit` only (read/glob/grep excluded per project scope). |
| IPC channel for file operation events | The renderer cannot see main-process events without a channel. Every existing real-time feature in this app (search progress, contact extraction, video download) uses a dedicated IPC channel. | Low | New channel `AI_FILE_OPERATION` in `channellist.ts`. Main-to-renderer direction only. Matches `CONTACT_EXTRACTION_PROGRESS` pattern. |
| Frontend subscription wrapper | The Vue layer needs a typed `windowReceive()` call matching the existing pattern in `src/views/api/aiChat.ts`. Without it, each component would duplicate raw IPC listener code. | Low | Add `subscribeToFileOperations()` and `unsubscribeFromFileOperations()` in `src/views/api/aiChat.ts` or a new `src/views/api/fileOperations.ts`. |
| In-chat operation badge | Users expect to see what the AI did inline in the conversation -- this is how Cursor, Windsurf, and Copilot Chat all work. A separate panel would be invisible. | Medium | Extend existing `TOOL_RESULT` message rendering in `AiChatBox.vue`. When `metadata.toolName` is `file_write` or `file_edit`, show a styled badge with file path, operation icon, and success/failure indicator. Reuse existing `v-chip` and `v-icon` Vuetify patterns. |
| Success and failure recording | Users need to know when the AI tried and failed, not just when it succeeded. Failed writes are common (path validation errors, permission denied, file exists in create mode). | Low | Emit record on both branches in `executeFileTool()`. The `FileToolService.execute()` return already includes `success: boolean` and optional `error: string`. |
| 6-language translations | Product ships with en, zh, es, fr, de, ja. Any untranslated user-facing string breaks the experience for 5 of 6 locales. | Medium | Add `fileOperations` section to all 6 lang files. Keys needed: `file_created`, `file_overwritten`, `file_edited`, `file_operation_failed`, `n_replacements`, `bytes_written`. Follow existing `skills.*` namespace pattern. |

## Differentiators

Features that elevate the UX beyond the minimum. Not expected, but valued by power users who care about auditability.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Expandable diff preview on edit | When the AI edits a file, users want to see what changed, not just that something changed. The `FileEditResult` already includes a `diff` field. | Medium | Collapsible `<details>` element inside the operation badge, showing unified diff lines (already generated by `diff.createPatch` in `FileToolService`). Green/red color-coded lines. Low cost because the diff data is already computed. |
| Operation summary counter | During a multi-step AI task, users want a quick tally: "3 files created, 1 edited". A badge counter in the chat header (next to the existing tool icons) provides this at a glance. | Low | Reactive `computed` counting records by type. Display as a small `v-badge` overlay on the existing tool icon. |
| Click-to-open-file from badge | Users see the AI changed `config.json` and want to open it immediately. Clicking the badge opens the file in the system editor. | Low | `shell.openPath(filePath)` via Electron shell API. Add `openFileInEditor` IPC handler (single line). Badge has `@click` handler. |
| Grouped operation display | When the AI performs 5 file writes in one turn, showing 5 separate badges clutters the chat. Grouping them under a collapsible "N file operations" summary keeps the conversation scannable. | Medium | Detect consecutive file operation messages in `AiChatBox.vue`. Replace with a single expandable group card. Requires tracking operation count per assistant message. |
| Operation timestamp in badge | Users want to know when the change happened relative to the conversation. "2 seconds ago" or absolute time helps with mental model. | Low | Reuse existing `formatTimestamp()` utility already in `AiChatBox.vue`. Add a small `<span>` in the badge footer. |

## Anti-Features

Features to explicitly NOT build. Each has been evaluated and rejected for v1.1.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Database persistence of operation records | Adds Entity, Model, Module, migration, and IPC handler for data that users only need during the current session. Premature complexity. | In-memory storage in `FileOperationTracker` (Map keyed by conversationId). Records lost on restart. If users demand history, add persistence in a future milestone. |
| Full rollback/undo system | Requires storing file content snapshots before each mutation, implementing a reverse-apply engine, handling edge cases (concurrent edits, deleted files). Massive scope creep for v1.1. | Show what changed (badge + optional diff). Let users manually undo or use version control. Rollback is a separate milestone if demanded. |
| Recording read-only operations (file_read, glob_files, grep_files) | These do not mutate anything. Recording them would flood the operation feed with noise and train users to ignore the badges. | Only record mutation tools: `file_write`, `file_edit`. Delete operations if added later. |
| Separate operation history panel/tab | Adds a new page, routing, and navigation. Splits attention between chat and panel. The chat context IS the right place to show what the AI did inline. | Inline badges within the existing chat message stream. Future: add a "session log" view if power users demand it. |
| View-diff for write (not edit) operations | `file_write` with `mode: "create"` has no previous content to diff. Showing "created new file" is sufficient. For overwrite mode, the previous content is not captured (FileToolService does not snapshot before write). | Show operation type badge ("Created" vs "Overwritten"). If diff-on-overwrite is needed later, modify `FileToolService` to snapshot before write. |
| Real-time streaming of file content as it is written | File writes in this app are atomic (`write-file-atomic`). There is no partial-write state to stream. Faking it would mislead users. | Show a single badge when the write completes. No intermediate states. |
| Rate limiting or debouncing of record emission | File tool calls are already rate-limited in `ToolExecutor` (see `FILE_TOOL_RATE_LIMITS`). Adding another layer of debouncing on the recording path adds complexity with no benefit. | Emit one record per tool call, immediately after execution. The upstream rate limiter already prevents floods. |

## Feature Dependencies

```
FileOperationRecord type (entityTypes/)
    |
    v
FileOperationTracker service (service/)
    |
    +--- ToolExecutor.executeFileTool() integration
    |        |
    |        v
    |    AI_FILE_OPERATION IPC channel (config/channellist.ts)
    |
    v
Frontend subscription wrapper (views/api/)
    |
    v
AiChatBox.vue badge rendering
    |
    v
6-language translations (views/lang/*.ts)
```

Dependency chain rationale:
- The type must exist first because every other layer references it.
- The tracker service depends on the type and produces records.
- ToolExecutor integration depends on the tracker (calls `tracker.emit()`).
- The IPC channel is required for tracker-to-renderer communication.
- The frontend subscription wrapper depends on the IPC channel existing.
- Badge rendering depends on both the subscription wrapper and the translations.
- Translations have no upstream dependency but must be completed before the feature ships.

## MVP Recommendation

Prioritize:
1. **FileOperationRecord type** -- foundational, blocks everything else
2. **FileOperationTracker service** -- core recording logic
3. **AI_FILE_OPERATION IPC channel** -- enables communication
4. **ToolExecutor integration** -- wires recording into existing tool dispatch
5. **In-chat operation badge** -- user-visible output
6. **6-language translations** -- shipping requirement

Defer:
- **Expandable diff preview**: Medium complexity, adds significant value but the diff data is already available in `FileEditResult.diff`. Easy to add after MVP. Low risk of architecture change.
- **Click-to-open-file**: Nice-to-have, single IPC handler, but not critical for visibility.
- **Grouped operation display**: Requires non-trivial rendering logic in AiChatBox.vue. Defer until usage patterns show whether clutter is actually a problem.
- **Operation summary counter**: Low cost but depends on grouped display to be useful.
