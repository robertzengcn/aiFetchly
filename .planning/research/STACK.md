## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| Event emission | IPC `event.sender.send()` (via callback) | Node.js EventEmitter + separate IPC forward | Adds indirection; the `event` reference is already threaded through `StreamEventProcessor` |
| State management | `ref<T[]>()` in AiChatBox or composable | Dedicated Pinia store | Single-component consumer in v1.1; Pinia adds boilerplate for no benefit unless other components need access |
| Schema validation | TypeScript discriminated unions | Zod schema validation | Operation records are constructed internally (not from user input); no external trust boundary to validate |
| Badge rendering | Vuetify `v-chip` with color/variant | Custom CSS component | Vuetify already imported; custom CSS adds maintenance burden |
| Operation type | Enum (`FileOperationType`) | String literal union | Enum is more consistent with existing `MessageType`, `PlanStepStatus` patterns in `commonType.ts` |
| Tracker lifetime | Static class (like `RateLimiterManager`) | Singleton instance | Static class matches established pattern in ToolExecutor.ts (lines 61-102) |

## Integration Architecture

### Data Flow

```
AI Chat Stream (LLM response)
  |
  v
StreamEventProcessor.handleToolCall()
  |
  v
ToolExecutor.executeInternal()
  |
  +---> case "file_write" / "file_edit":
  |       executeFileTool() --> FileToolService.execute()
  |       result returned
  |       |
  |       +---> FileOperationTracker.emit(record, sender)
  |             |
  |             v
  |         sender.send(AI_FILE_OPERATION, JSON.stringify(record))
  |             |
  |             v
  |         Renderer: windowReceive(AI_FILE_OPERATION, callback)
  |             |
  |             v
  |         AiChatBox.vue: fileOperations ref updated
  |
  v
StreamEventProcessor continues processing
```

### Key Integration Points

1. **ToolExecutor.executeFileTool()** (line 1319-1327 in ToolExecutor.ts)
   - Current: calls `FileToolService.execute()` and returns result directly
   - Change: wrap the call, inspect `toolName` and `result.success`, emit record for write-like operations
   - The `conversationId` is available from `execute()` (line 111-131) but NOT passed to `executeFileTool()` currently
   - Must thread `conversationId` through to `executeFileTool` or pass it to `FileOperationTracker.emit()`

2. **StreamEventProcessor event reference** (line 220: `this.event.sender.send()`)
   - The `IpcMainEvent` is held in `StreamEventProcessor.state` or passed to tool execution
   - `FileOperationTracker.emit()` needs the sender reference to push to renderer
   - Solution: pass `sender` callback or the event itself to `executeFileTool()`, or have `FileOperationTracker` hold a registered sender

3. **Channel registration** (src/config/channellist.ts)
   - Add: `export const AI_FILE_OPERATION = "ai-chat:file-operation";`
   - Follows existing pattern: `AI_CHAT_STREAM_CHUNK`, `AI_CHAT_STREAM_COMPLETE`

4. **Frontend subscription** (src/views/api/aiChat.ts)
   - Add `onFileOperation()` function using `windowReceive(AI_FILE_OPERATION, callback)`
   - Mirror existing pattern: `streamChatMessage()` at line 117-191

5. **AiChatBox.vue** (src/views/components/aiChat/AiChatBox.vue)
   - Subscribe to `AI_FILE_OPERATION` in `onMounted` / cleanup in `onUnmounted`
   - Display operation badges inline with tool call messages
   - Use Vuetify `v-chip` with color-coded variants per operation type

### FileOperationTracker Design

The tracker should follow the `RateLimiterManager` pattern (static class, no instantiation needed):

```typescript
// Pseudocode for src/service/FileOperationTracker.ts
export class FileOperationTracker {
  private static sender: IpcMainEventSender | null = null;

  static registerSender(sender: IpcMainEventSender): void { ... }

  static emit(record: FileOperationRecord): void {
    try {
      this.sender?.send(AI_FILE_OPERATION, JSON.stringify(record));
    } catch {
      // Silently swallow -- tracking must never break tool execution
    }
  }
}
```

The `sender` is registered once when a stream starts (in `StreamEventProcessor` constructor or `handleToolCall`) and cleared when the stream completes.

### FileOperationRecord Type

```typescript
// Pseudocode for addition to src/entityTypes/fileToolTypes.ts
export enum FileOperationType {
  CREATE = "file_create",
  OVERWRITE = "file_overwrite",
  EDIT = "file_edit",
  DELETE = "file_delete",
}

export interface FileOperationRecord {
  readonly id: string;              // UUID or timestamp-based
  readonly type: FileOperationType;
  readonly filePath: string;
  readonly timestamp: number;       // Date.now() for IPC-safe serialization
  readonly success: boolean;
  readonly conversationId: string;
  readonly toolCallId?: string;     // Links to the tool call in chat
  readonly bytesWritten?: number;   // For create/overwrite
  readonly replacements?: number;   // For edit
  readonly error?: string;          // If failed
}
```

No `Date` objects -- use `number` (epoch ms) for clean IPC serialization. The `toolCallId` links the record back to the specific tool call chunk already displayed in the chat.

## What NOT to Add

| Do Not Add | Reason |
|------------|--------|
| Any new npm package | Zero new dependencies needed; all capabilities exist |
| Database persistence (TypeORM entity) | Explicitly out of scope per PROJECT.md; in-memory only in v1.1 |
| Pinia store for file operations | Over-engineering for single-component consumer; use `ref` |
| Zod/joi validation | Records are constructed internally, not from external input |
| WebSocket or SSE | IPC already handles real-time push; no network layer needed |
| Debounce/throttle library | Operation events are infrequent (one per file tool call) |
| UUID library (`uuid` npm package) | Use `crypto.randomUUID()` (available in Electron/Node 19+) or `${Date.now()}-${Math.random().toString(36).slice(2)}` |
